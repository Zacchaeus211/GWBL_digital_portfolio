01 — Onboarding & Compliance Training

What I learned

How a bank the size of OCBC actually operates day to day — how Group O&T splits work between running the bank and changing it, and why compliance rules like PDPA and insider trading policy exist before any technical work even starts.

Applying KSA

Attitude and awareness, more than a hard skill at this stage: understanding the risk and regulatory context I'd be operating in, so I knew where the lines were once I started touching real data.

Growth

Gave me the operating context for everything that followed — the confidentiality boundaries I've applied throughout this portfolio come directly from this training.

02 — Composite View Across Multiple Products

What I learned

How to design a SQL view that pulls consistent, standardized output from several differently-structured source tables at once — handling date parsing, business-rule mapping via CASE logic, and multiple join conditions in one query.

Applying KSA

Applied SQL fundamentals I'd studied — joins, CASE statements, date functions — to a real, messy dataset for the first time, where the "right" answer isn't in a textbook but has to match how the business actually defines each product.

Growth

Built my confidence in reading and reasoning about production-scale queries, and taught me to think about a query as documentation of business logic, not just as code.

03 — Dashboard Data Pull

What I learned

How to combine data from multiple source systems — some live, some decommissioned — into a single clean base table using UNION ALL, and how to reconcile schema differences between them.

Applying KSA

Practiced writing SQL for a real reporting use case rather than an isolated exercise, and learned to plan a data model ahead of the visualization it will eventually support.

Growth

The visualization side is on hold because there isn't enough data yet — a good early lesson that a dashboard is only as useful as the pipeline feeding it, and that "not ready yet" is a legitimate project status, not a failure to report around.

04 — Power BI → Data Lake Migration (XMapping)

What I learned

How an existing dashboard's data source gets re-pointed from a legacy system to an Enterprise data lake on Hadoop — including creating new backing tables, rewriting Power Query steps, and cleaning source data so the numbers still tie out after the switch.

Applying KSA

Combined data engineering (SQL table creation, source data cleaning) with reporting tooling (Power Query, and a first introduction to how the resulting visuals get built), plus documentation skills — writing steps clear enough for another user to run the weekly refresh without me.

Growth

This is where I started to see data governance and data visualization as connected: a dashboard's trustworthiness depends on the governance decisions made upstream, long before anyone opens the report.

05 — AD Group Access Matrix

What I learned

How access to systems and data is actually controlled behind the scenes — mapping Active Directory groups to what they should and shouldn't be able to see or do.

Applying KSA

Direct exposure to data governance as a discipline, working alongside my supervisor rather than independently — an area I hadn't studied formally before this internship.

Growth

Reframed how I think about "who can see the data" as its own engineering problem, separate from but just as important as building the pipeline or the dashboard itself.

Overall Growth

Coming into this internship, my exposure to SQL, data governance, and data visualization was mostly academic. Across these few weeks I've moved from learning how OCBC's Group Operations & Technology division is structured, to writing SQL against real production-shaped data, to working on an active migration that touches data engineering, reporting, and governance at once. The through-line for me has been realizing how much of "good data work" happens before a chart is ever drawn — in the compliance training, the table design, and the access control decisions that make the eventual dashboard trustworthy. That's shifted how I think about my direction in IT: I'm more drawn to the data engineering and governance side of the stack than I expected to be going in.
