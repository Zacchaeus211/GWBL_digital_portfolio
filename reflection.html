01 — Onboarding & Compliance Training

What I learned

How a bank the size of OCBC actually operates day to day — how Group O&T splits work between running the bank and changing it, and why compliance rules like PDPA and insider trading policy exist before any technical work even starts.

Applying KSA

Attitude and awareness, more than a hard skill at this stage: understanding the risk and regulatory context I'd be operating in, so I knew where the lines were once I started touching real data.

Growth

Gave me the operating context for everything that followed — the confidentiality boundaries I've applied throughout this portfolio come directly from this training.

02 — Composite Views Across 12 Products

What I learned

I'd learned SQL joins in school, but here I learned what makes a join actually safe — which columns can be joined on without silently duplicating rows, how to sift through tables with hundreds of columns to find the right join keys, and how to troubleshoot when a join quietly breaks the row count.

Applying KSA

School gave me the syntax — how to write a JOIN, a CASE statement, a date function. It never taught me how to go looking for the right join condition across unfamiliar tables, or how to document that reasoning so someone else could follow it. That part I had to build from scratch, across all 12 product views.

Growth

This is where I stopped thinking of SQL as "a subject I passed" and started thinking of it as a tool I still had a lot to learn about. Writing one view that a whole department could rely on, instead of a query for a single exercise, changed how carefully I check my own joins now.

03 — Dashboard Data Pull

What I learned

How to combine data from multiple source systems — some live, some decommissioned — into a single clean base table using UNION ALL, and how to reconcile schema differences between them.

Applying KSA

Practiced writing SQL for a real reporting use case rather than an isolated exercise, and learned to plan a data model ahead of the visualization it will eventually support.

Growth

The visualization side is on hold because there isn't enough data yet — a good early lesson that a dashboard is only as useful as the pipeline feeding it, and that "not ready yet" is a legitimate project status, not a failure to report around.

04 — XMapping: Source CSV → Data Lake → Power BI

What I learned

That re-pointing a data source is never just re-pointing it. The old process needed a person to drop a file into Teams and manually hit refresh every time; before I could automate that away, I had to read through the department's existing Python scripts to understand exactly which rows they filtered out of the dashboard and why, then replicate that logic faithfully in Power Query — on top of the cleaning, table design, and documentation work underneath. The refresh itself is automated now; automating the cleaning step is what I'm working on next.

Applying KSA

School's Python basics made reading their scripts manageable, which made the harder part — translating conditional logic across two different languages without changing the result — actually achievable. Combined that with SQL table design and Power Query automation, plus documentation clear enough for someone else to run the refresh.

Growth

I've completed one dashboard so far, with several more still queued — and each one will need its own approach, since no two departments clean or filter their data the same way. That's taught me not to expect a single reusable fix, and to budget the "understand the existing logic" step as seriously as the build itself.

05 — AD Group Access Matrix

What I learned

That a pipeline isn't finished once the data flows correctly — it also needs to be locked down correctly. Since each department owns its own pipeline and its own dashboard, the access matrix has to keep both scoped: a department can't see another's underlying data, and can't view another's dashboard either.

Applying KSA

Direct exposure to data governance as a discipline, working alongside my supervisor rather than independently — this is genuinely new ground for me, nothing I'd covered in school before this internship.

Growth

Reframed how I think about "who can see the data" as its own engineering problem, tied directly to the pipeline I was building — not a separate, later concern, but something that has to be designed alongside it.

Overall Growth

Coming into this internship, I thought I already knew SQL — school had covered joins, CASE statements, the basics. What this internship showed me is that the basics are just the starting point. School never taught me how to sift through tables with hundreds of columns to find a safe join key, how to document that reasoning for someone else to follow, or how to think through refresh cadence and partitioning before a table even exists. Data governance — deciding who in Operations should be able to see what — was something I hadn't touched in school at all. The through-line for me has been realizing how much of "good data work" happens before a chart is ever drawn: in the join logic, the table design, and the access control decisions that make the eventual dashboard trustworthy. That's shifted how I think about my direction in IT — I'm more drawn to the data engineering and governance side of the stack than I expected to be going in. My work so far has stayed on the SQL and Power Query side of things; the dashboards I've helped re-route still need their visualization layer built out, which is the direction I expect to grow into next.
