02 — Composite Views Across 12 Products

status: live (sanitized)

Weeks 3–8 · SQL (Hive/Spark SQL)

The department needed one place to query all of their data, instead of pulling different tables for different products every time. I built composite views for 12 different products, each consolidating fields from several source tables — some with hundreds of columns — into a single standardized output view, with date fields normalized, category codes mapped from raw system codes to readable business labels, and consistent column naming across all 12 views so they could eventually sit side by side.

The hard part wasn't writing the SQL — it was figuring out how to join. For each product I had to sift through the source tables to work out which columns were safe to join on without silently duplicating rows, then document that reasoning so the join logic wasn't just sitting in the query, it was traceable. One representative view is shown below; the other 11 follow the same approach against different source tables. Table and column names have been genericized — the underlying logic and structure are unchanged. 

with temp as
(
select distinct
 TABLE_1.COLUMN_1 as OUTPUT_COLUMN_1
,CAST(FROM_UNIXTIME(UNIX_TIMESTAMP(CASE WHEN TABLE_1.COLUMN_2 = 'CASE_VALUE_1' THEN NULL ELSE TABLE_1.COLUMN_2 END,'yyyyMMdd')) AS DATE) AS OUTPUT_COLUMN_2
,TABLE_1.COLUMN_3 as OUTPUT_COLUMN_3
,CASE
WHEN TABLE_1.COLUMN_4 = 'CASE_VALUE_2' THEN 'CASE_RESULT_1'
WHEN TABLE_1.COLUMN_4 = 'CASE_VALUE_3' THEN 'CASE_RESULT_2'
ELSE 'CASE_DEFAULT_1' END as OUTPUT_COLUMN_7
,TABLE_2.COLUMN_6 as OUTPUT_COLUMN_9
,TABLE_3.COLUMN_8 as OUTPUT_COLUMN_11
,TABLE_4.COLUMN_12 as OUTPUT_COLUMN_16
-- ... additional standardized / mapped columns omitted for length ...
,'BG' as OUTPUT_COLUMN_84
from
(
  select COLUMN_1, COLUMN_2, COLUMN_3, COLUMN_4, COLUMN_5, COLUMN_7, COLUMN_9,
         COLUMN_14, COLUMN_15, COLUMN_16, COLUMN_17
  from TABLE_1
  where FILTER_COLUMN_1 = '9999-12-31'
) TABLE_1
left join
(
  select distinct COLUMN_6, COLUMN_7, COLUMN_31, COLUMN_32, COLUMN_33
  from TABLE_2
  where FILTER_COLUMN_1 = '9999-12-31' and COLUMN_33 = '1'
) TABLE_2
  on TABLE_1.COLUMN_1 = TABLE_2.COLUMN_31
 and TABLE_1.COLUMN_7 = TABLE_2.COLUMN_7
left join
(
  select COLUMN_8, COLUMN_10, COLUMN_11, COLUMN_34, COLUMN_7, COLUMN_9, COLUMN_22, COLUMN_23
  from TABLE_3
  where FILTER_COLUMN_1 = '9999-12-31'
) TABLE_3
  on TABLE_1.COLUMN_7 = TABLE_3.COLUMN_7
left join
(
  select COLUMN_13, COLUMN_12
  from TABLE_4
  where FILTER_COLUMN_1 = '9999-12-31'
) TABLE_4
  on TABLE_3.COLUMN_34 = TABLE_4.COLUMN_13
)
select * from temp;

03 — Data Pull for Reporting Dashboard

status: live (sanitized)

Weeks 3–8 · SQL · Visualization not yet started (pending data)

Wrote a script to pull and standardize data from eight source tables (a mix of live and decommissioned systems) via UNION ALL, tagging each row with its originating source, then joined a reference table on a trimmed key to enrich the output with additional date fields. This forms the base dataset for a dashboard I'm building — visualization work is paused until the source data volume is sufficient. 

WITH CTE_1 AS
(
-- SOURCE 1 - LIVE
SELECT 'MASKED_VALUE_1' AS OUTPUT_COLUMN_1, CAST(COLUMN_1 AS STRING) AS OUTPUT_COLUMN_2,
       COLUMN_2 AS OUTPUT_COLUMN_3, COLUMN_3 AS OUTPUT_COLUMN_4 /* ...+17 more columns */
FROM TABLE_1
WHERE FILTER_COLUMN_1 = 'FILTER_VALUE_1'

UNION ALL

-- SOURCE 2 - DEAD
SELECT 'MASKED_VALUE_2' AS OUTPUT_COLUMN_1, CAST(COLUMN_1 AS STRING) AS OUTPUT_COLUMN_2,
       COLUMN_2 AS OUTPUT_COLUMN_3, COLUMN_3 AS OUTPUT_COLUMN_4 /* ... */
FROM TABLE_2
WHERE FILTER_COLUMN_1 = 'FILTER_VALUE_1'

-- ... 6 more UNION ALL blocks across sources 3–8, alternating LIVE / DEAD systems ...
),

CTE_2 AS
(
SELECT CAST(COLUMN_1 AS STRING) AS OUTPUT_COLUMN_2, COLUMN_21 AS OUTPUT_COLUMN_22,
       COLUMN_22 AS OUTPUT_COLUMN_23, COLUMN_23 AS OUTPUT_COLUMN_24
FROM TABLE_9
WHERE FILTER_COLUMN_1 = 'FILTER_VALUE_1'
)

SELECT CTE_1.*, CAST(CTE_2.OUTPUT_COLUMN_22 AS DATE) AS OUTPUT_COLUMN_22,
       CTE_2.OUTPUT_COLUMN_25, CTE_2.OUTPUT_COLUMN_26
FROM CTE_1
LEFT JOIN CTE_2
  ON TRIM(CTE_1.OUTPUT_COLUMN_2) = TRIM(CTE_2.OUTPUT_COLUMN_2);

04 — XMapping: Source CSV → Data Lake → Power BI

status: live (sanitized)

Ongoing track · SQL, Power Query (M), Python (reference), documentation

Previously, keeping one of these dashboards up to date was a manual process: someone had to place the source file into a Teams folder, Power BI's source was pointed at that Teams file, and someone had to manually hit refresh every time. XMapping automates the refresh step — by re-pointing Power BI at the Enterprise data lake instead of the Teams file, the dashboard can refresh on a schedule with no one needing to click anything. Each dashboard is owned by a different department, with its own WFTS folder feeding its own table — so this isn't one big shared pipeline, it's many separate ones, each following the same general shape but built and owned independently. Departments can only see their own pipeline and their own dashboard, not each other's.

The refresh itself is now automated; what's still manual — and what we're actively working on automating next — is the preprocessing: cleaning and standardizing the source data before it's fit for the data lake table. The goal is to move that cleaning step into Power BI itself, so it happens automatically as part of the refresh rather than as a separate manual task (see Power Query snippet 1, below).

One instance of this pattern, per dashboard — I've completed one so far; several more are still queued and each will need its own version of this, since no two dashboards clean, filter, or refresh their data the same way.

My part of this covers:

  • Understanding the dashboard's existing update logic before touching anything — the department sent over the Python scripts they used to filter and prepare the data beforehand. I had to read through their filtering conditions (the specific rows they excluded from ever reaching the dashboard, and why) before I could touch the pipeline at all — this wasn't just swapping a data source.
  • Replicating that filtering logic in Power Query, so the new pipeline behaves identically to the old one from the department's point of view, and automating it so it runs on schedule without needing the manual Python step each time.
  • Cleaning the source CSV to match the exact format the data lake table expects, then documenting the cleaning steps and the do's and don'ts a user must follow when uploading data into the WFTS folder — so the format stays standardized every time, regardless of who's uploading.
  • Designing the data lake table — before creating it I had to work through questions like: is this data refreshed monthly? Should new data be appended, or does the table get fully dumped and replaced each time? Those answers shape how the table is partitioned, so it can be queried efficiently and cleanly later on.
  • Writing refresh documentation so other users can run the dashboard's weekly data refresh themselves, without needing me involved each time.

My work on this track has been on the SQL and Power Query side so far; the data visualization side of these dashboards is something I expect to pick up more of going forward.

Note: the SQL scripts used to create the data lake table, the original Python filtering scripts, and the cleaning / upload documentation are sensitive and cannot be shown. The two Power Query transformations below — the parts of this work I'm able to share — are sanitized.

Power Query — Type Casting & Data Cleaning

This is part of the effort to move preprocessing into Power BI itself — standardizing column types, parsing dates across multiple possible formats, and clearing out error values, so this cleaning happens automatically on refresh instead of as a manual step beforehand. 

let
 STEP_1 = Odbc.Query(
 PARAMETER_1,
 if Text.From(PARAMETER_2) = "MASKED_VALUE_1" then
   "select * from MASKED_DATABASE.""MASKED_SCHEMA"".MASKED_TABLE limit 0"
 else
   "select * from MASKED_DATABASE.""MASKED_SCHEMA"".MASKED_TABLE where FILTER_COLUMN_1 = 'FILTER_VALUE_1'"
 ),

 STEP_2 = Table.TransformColumns(STEP_1, {
   { "COLUMN_1", each
       let
         VARIABLE_1 = _,
         VARIABLE_2 = if VARIABLE_1 = null then null else Text.Trim(Text.From(VARIABLE_1)),
         VARIABLE_3 =
           if VARIABLE_1 = null or VARIABLE_2 = "" then null
           else try Date.From(DateTime.FromText(VARIABLE_2, "en-GB"))
             otherwise try Date.From(DateTime.FromText(VARIABLE_2, "en-SG"))
             otherwise try Date.From(DateTime.FromText(VARIABLE_2, "en-US"))
             otherwise null
       in VARIABLE_3,
     type date }
 }),

 STEP_3 = Table.TransformColumnTypes(STEP_2, {
   {"COLUMN_2", Int64.Type}, {"COLUMN_4", type text}, {"COLUMN_1", type date}
   /* ...remaining column type declarations... */
 }),

 STEP_4 = Table.RemoveColumns(STEP_3, {"COLUMN_19"}),

 STEP_5 = Table.ReplaceErrorValues(STEP_4, {
   {"COLUMN_2", null}, {"COLUMN_4", "MASKED_VALUE_2"}, {"COLUMN_1", null}
   /* ...remaining fallback values... */
 }),

 STEP_8 = Table.SelectRows(STEP_7, each
   not List.IsEmpty(List.RemoveMatchingItems(Record.FieldValues(_), {"", null}))
 ),

 STEP_10 = Table.AddColumn(STEP_9, "COLUMN_24", each
   if [COLUMN_13] = null then "MASKED_RESULT_1"
   else if Text.StartsWith([COLUMN_13], "MASKED_PREFIX_1") then "MASKED_RESULT_2"
   else if Text.StartsWith([COLUMN_13], "MASKED_PREFIX_4") then "MASKED_RESULT_3"
   else [COLUMN_13],
   type text)
in
 STEP_10

Power Query — Replicating the Department's Python Filtering Logic

This part started with the department's own Python scripts, not mine. They filter out rows they don't want reaching the dashboard, based on a set of business rules — so before writing anything, I read through their Python to understand exactly which rows got excluded and why. I'd learned Python basics in school, which made reading their logic manageable; the harder part was translating conditional filtering logic from Python into Power Query's M language faithfully enough that the output matched, then automating it so it runs on the dashboard's schedule without manual intervention. That's also why this step chain runs over 20 steps — each rule from their Python script became its own traceable transform here, rather than one black-box condition. 

let
 STEP_1 = Odbc.Query(PARAMETER_1, "select * from MASKED_DATABASE.""MASKED_SCHEMA"".MASKED_TABLE ..."),
 STEP_2 = Table.PromoteHeaders(STEP_1, [PromoteAllScalars=true]),
 STEP_3 = Table.TransformColumnTypes(STEP_2, { {"COLUMN_1", Int64.Type}, {"COLUMN_6", type date} /* ... */ }),
 STEP_4 = Table.AddIndexColumn(STEP_3, "COLUMN_25", 0, 1, Int64.Type),

 -- Derive normalized lookup flags from raw fields
 STEP_9 = Table.AddColumn(STEP_8, "COLUMN_30", each
   if [COLUMN_4] = null then "" else Text.Upper(Text.Trim(Text.From([COLUMN_4]))), type text),

 -- Categorize each row against defined business-rule reference lists
 STEP_13 = Table.AddColumn(STEP_12, "COLUMN_34", each
   if Text.Contains([COLUMN_26], "MASKED_VALUE_4") and Text.Contains([COLUMN_26], "MASKED_VALUE_2")
     then "MASKED_RESULT_3"
   else if Text.Contains([COLUMN_26], "MASKED_VALUE_5") and Text.Contains([COLUMN_26], "MASKED_VALUE_3")
     then "MASKED_RESULT_6"
   else "MASKED_RESULT_7",
   type text),

 STEP_14 = Table.SelectRows(STEP_13, each
   List.Contains({"MASKED_RESULT_3","MASKED_RESULT_4","MASKED_RESULT_5","MASKED_RESULT_6"}, [COLUMN_34])
 ),

 -- Apply status + reason-code logic per business rule, used to flag records for the weekly refresh
 STEP_18 = Table.AddColumn(STEP_17, "COLUMN_38", each
   if [COLUMN_34] = "MASKED_RESULT_3" and [COLUMN_32] = true and [COLUMN_28] = "MASKED_VALUE_6"
     then "MASKED_STATUS_1"
   else "MASKED_STATUS_2",
   type text),

 STEP_20 = Table.RemoveColumns(STEP_19, {"COLUMN_37","COLUMN_32","COLUMN_26" /* ...+more */}, MissingField.Ignore)
in
 STEP_20

05 — Active Directory Group Access Matrix

status: restricted

Data Governance · tied to the XMapping project (#04)

Worked with my supervisor to build an AD group access matrix specifically governing access to one department's XMapping pipeline (#04) — its source CSV, its WFTS folder, its data lake table, and its Power BI refresh. Since each department owns its own pipeline and its own dashboard, access needs to be scoped per department on both fronts: a department can't see another department's underlying pipeline, and can't view another department's dashboard either. The matrix maps which Active Directory groups get access to which stage.

This was genuinely new ground for me — access control and data governance at this level isn't something covered in school, so this was learned entirely on the job, working directly alongside my supervisor.

Note: this item is described in writing only. The matrix contains internal access-control mappings that fall outside what can be shared publicly, in line with data governance requirements.
